# ポン読解(仮)— 構想整理と実装計画

作成日: 2026-07-08
対象: 「開いてボタンを押すだけで英文解釈教材が一瞬で出る」隙間時間学習アプリのMVP計画。

---

## 1. このアイデアの要約

**「AI教材チャット」ではなく「AIが書いた英文解釈問題集」を作る話である。**

構想の本質は次の一行に集約できる:

> リアルタイムAI生成アプリではなく、**AIをビルド時のコンテンツ工場として使う、即時表示型の問題集アプリ**。

- ユーザー体験: 難易度とテーマを選ぶ → 0秒で長文+和訳+構文+語彙+設問が出る
- AIの位置づけ: ランタイムには存在しなくてよい。事前生成パイプラインの中にだけいる
- 競合はChatGPTではなく、「ポラリス」「やっておきたい英語長文」「スタサプ」などの既存教材・アプリ

「アプリ開発かAI自動化ツールか」という問い(相談事項1)への答えは明確に**アプリ開発**。AIは教材の著者であり、アプリの機能ではない。この整理ができた時点で、MVPからAI接続を切り離せる。

## 2. 使える部分(強い点)

1. **「プロンプトを書かせない」という割り切り。** ChatGPTとの差別化はここにしかなく、そこを最初から中心に置けている。
2. **事前生成+キャッシュ方式(相談事項4)。** 現実的どころか、これが製品そのもの。副次効果が大きい:
   - 表示遅延ゼロ(「ポン」体験の実現手段はこれしかない)
   - コストがユーザー数に比例しない(生成数に比例する)
   - **出す前に人間が検品できる** — 品質担保の唯一の現実解
3. **構造化JSONで返させる方針。** 表示・保存・既読管理すべてが楽になる。
4. **タブ式解説パネル。** 「まず英文だけ読む → 詰まったら構文タブ」という読解訓練の正しい順序をUIが強制できる。ChatGPTだと和訳が先に目に入ってしまう。
5. **既存資産。** このリポジトリには既に localStorage ベースの既読管理・連続記録・XP の実装パターン(はなす日記)がある。継続動機の仕組みを流用できる。

## 3. 弱い部分

1. **構文解説の正確性が生命線なのに、AIはそこを間違える。** 受験生は解説の誤りに極めて敏感で、1回「この構文解説おかしい」と思われたら信頼は戻らない。和訳の自然さより構文ラベル(SVOC・関係詞・分詞構文)の誤りが致命傷になる。
2. **継続利用の理由が「無限に問題が出る」だけでは弱い。** 問題が無限に出ること自体はChatGPTでもできる。3日で飽きる典型パターン。
3. **「タブレット横画面」と「隙間時間」は矛盾しがち。** 隙間時間の実態はスマホ縦画面。横2ペイン前提で作ると、一番使ってほしい場面で使えないアプリになる。
4. **テーマ選択は初期にはほぼ価値がない。** 教材プール100題の段階で「経済×難関大」のような掛け算をすると各セルが数題しかなく、すぐ枯渇する。
5. **差別化が最終的に「教材の質」勝負になる。** そこはプロ予備校講師が書いた既存教材が最強で、個人開発が質で勝つのは難しい。勝てるのは「即時性 × 無限供給 × 自分の弱点に合う」の掛け算だけ。

## 4. 反対意見(あえて潰しにいく)

- **「ChatGPTでよくないか」** — 正直、カスタム指示を1回書けば体験の7割は再現できる。残り3割(起動即表示・タブUI・既読管理・検品済み品質)に価値を感じる人だけがユーザー。つまり「AIで教材を作れること」は売りにならない。売りは**摩擦ゼロと品質保証**。
- **「それは問題集アプリでは」** — その通りで、それでいい。「AIアプリを作っている」という自意識を捨てて「更新頻度が無限の問題集」と割り切った方が設計がぶれない。
- **「品質検品に人手がかかるなら自動化の意味がない」** — 半分正しい。ただし検品は「全文添削」ではなく「構文ラベルの誤りチェック」に絞れば1題2〜3分。100題でも数時間。個人開発の初期規模なら成立する。
- **「オンデバイスAIは?」** — MVPでは論外。事前生成方式ならランタイムAIすら不要なので、この論点自体が消える。

## 5. 現実的なMVP案

**MVP = 「検品済み20題が入った、即表示のタブ式読解アプリ」。AI接続コードはアプリに1行も入れない。**

### 入れるもの
- 難易度選択(**高校基礎/高校標準/難関大 の3つだけ**。中学とTOEICは対象ユーザーが別人格なので後回し)
- テーマ選択は**「ランダム」固定で開始**(UIだけ置いてもよいが、フィルタ実装は題数50超えてから)
- 「ポンと出す」→ プールから未読1題を即表示
- 本文表示 + タブ(和訳/構文/語彙/設問)
- 設問の解答→正誤→解説表示
- 「もう一題」
- 既読管理・「読んだ題数」「連続日数」(localStorage、はなす日記の実装を流用)

### 入れないもの
- ログイン/課金/バックエンド/DB(教材は静的JSONでリポジトリに同梱)
- リアルタイムAI生成(生成は開発者が手元でスクリプト実行)
- SVOC色分け・音声・履歴分析

### 画面設計(相談事項8)
- 画面は実質2つ: **ホーム**(難易度選択+ポンと出すボタン+今日の記録)と**読解画面**
- 読解画面はレスポンシブ1画面:
  - スマホ縦: 上=本文スクロール、下=タブバー(タップで解説がボトムシートで出る)
  - タブレット横: 左=本文、右=タブ付き解説パネル(md:grid-cols-2)
- 「和訳タブは1文タップで該当文の訳だけ出す」は将来課題。MVPは全文一括でよい

## 6. 技術構成案(相談事項3,5,6)

**このリポジトリに新ルートとして追加する。新規プロジェクトは作らない。**

```
ランタイム:  Next.js(既存) + 静的JSON + localStorage — サーバー処理ゼロ、Vercel無料枠
生成側:     scripts/generate-material.ts(手元で実行するNodeスクリプト)
             → Gemini Flash or GPT-4o-mini に構造化出力で生成させる
             → content/materials/*.json に出力
             → 開発者が検品して git commit(検品=デプロイゲート)
```

- **外部API依存で問題ないか(相談事項3):** 問題ない。むしろ事前生成方式ならAPIが落ちてもアプリは無傷。ベンダーロックインも生成スクリプト1ファイルの書き換えで済む。
- **コスト(相談事項5):** Gemini Flash級なら1題あたり入力500+出力3,000トークン程度=約1円前後。**1,000題作っても千円台**。コストが問題になる構成はリアルタイム生成だけで、それはやらない。
- **Supabase/Workersはいつ要るか:** 「ユーザーごとの履歴同期」か「プール自動補充」が欲しくなった時。MVPでは不要。
- **APIキー:** フロントに置かない問題は、そもそもランタイムにAIがないので発生しない。生成スクリプトは `.env.local`(gitignore済み)から読む。

### データモデル
構想のJSONスキーマはほぼそのまま採用。変更点のみ:
- `id`(string)と `createdAt` を追加(既読管理に必須)
- `level` は `"basic" | "standard" | "advanced"` のコード値にし、表示名はUI側で持つ
- `answer` は選択肢の**インデックス(number)**にする("B"という文字列は選択肢シャッフルで壊れる)
- `sentenceExplanations[].structure` は自由文字列でよいが、生成プロンプト側でラベル語彙を制限する

## 7. 実装タスク分解(相談事項7)

依存順。各タスクは1コミット単位を想定。

1. **T1: 型定義とモック教材3題** — `src/types/reading.ts` + `content/materials/` に手検品済みJSON3題(1題は自分で全文検品する。これが品質基準のサンプルになる)
2. **T2: 読解画面** — `/reading` ルート、本文+タブUI、レスポンシブ。モック3題を表示
3. **T3: 設問インタラクション** — 選択→正誤→解説、1題内の複数設問対応
4. **T4: ホーム画面と出題ロジック** — 難易度選択、未読優先のランダム抽選、「もう一題」
5. **T5: 記録機能** — localStorage(既読ID、読了数、連続日数)。はなす日記のパターン流用
6. **T6: 生成スクリプト** — `scripts/generate-material.ts`。難易度・テーマ・題数を引数に取り、スキーマバリデーション(zod)を通ったものだけ出力
7. **T7: 一括生成+検品** — 各難易度7題ずつ計20題生成、人力検品、修正 or 破棄
8. **T8: デプロイ** — Vercel、スマホ/タブレット実機確認

T1〜T5はAI一切不要で完結する。ここまでで製品体験の検証ができる。

## 8. Claude Code / Codex に投げる実装指示文

そのまま貼れる形:

```
このNext.jsリポジトリ(App Router / Tailwind 4 / TypeScript)に英文解釈学習機能を追加してください。
注意: このリポジトリのNext.jsは新しいバージョンで慣習が異なる可能性があります。
実装前に node_modules/next/dist/docs/ の該当ガイドを確認してください。

## 作るもの
/reading 配下の「英文解釈問題集」機能。AI接続は不要。教材は静的JSONから読む。

## データ
- 型: src/types/reading.ts に ReadingMaterial を定義
  { id, title, level("basic"|"standard"|"advanced"), theme, estimatedMinutes,
    passage(string, 段落は\n\n区切り),
    sentenceExplanations: {sentence, structure, translation, commentary}[],
    vocabulary: {word, meaning, example}[],
    questions: {question, choices: string[4], answerIndex: number, explanation}[] }
- 教材: content/materials/*.json(モック3題を作成。中身は高校レベルの自然な英文で、
  構文解説は「SVOC/関係詞/分詞構文/仮定法/倒置」等の標準的な用語のみ使用)

## 画面
1. /reading — ホーム。難易度3択(高校基礎/高校標準/難関大)+「ポンと出す」ボタン
   +今日の記録(読了数・連続日数)
2. /reading/[id] — 読解画面。
   - スマホ縦: 本文の下に固定タブバー(和訳/構文/語彙/設問)、タップでボトムシート表示
   - md以上: 左に本文、右に解説パネル(タブ切替)の2カラム
   - 構文タブ: 一文ごとにカード表示(英文/構造ラベル/和訳/解説)
   - 設問タブ: 選択肢タップ→正誤表示→解説表示
   - 「もう一題」ボタン: 同難易度の未読からランダム遷移

## ロジック
- 出題: 選択難易度でフィルタ→localStorageの既読IDを除外→ランダム1題。
  全既読なら既読込みで抽選し「全問読了」バッジ表示
- 記録: localStorage(キー: pon-dokkai/v1)に 既読ID配列・読了日リストを保存。
  連続日数はクライアントで算出
- SSRとlocalStorageの不整合(hydration)に注意。記録表示はマウント後に読む

## 品質
- npm run lint / typecheck / test を通す
- 出題ロジック(フィルタ・未読優先)は vitest でユニットテストを書く
```

生成スクリプト(T6)は別タスクとして分けて投げる方が事故が少ない。

## 9. 最初の1週間でやるべきこと

- **Day 1–2:** T1+T2。モック3題のうち1題は自分で英文と構文解説を全文検品し「合格品質の見本」を作る
- **Day 3:** T3+T4(設問・出題ロジック)
- **Day 4:** T5(記録)+ Vercelデプロイ。**この時点で自分のスマホ/タブレットに入れて実際の隙間時間で使う**
- **Day 5:** T6(生成スクリプト)。見本1題をfew-shotとしてプロンプトに埋め込む
- **Day 6:** T7。20題生成→検品。**何%が検品を通るかを記録する**(この歩留まりがこの製品の成立可能性そのもの)
- **Day 7:** 検品済みプールで3セッション自習し、「ポン体験」が本当に気持ちいいか自己評価

### 1週間後の判断基準
- 生成歩留まりが5割を切る → プロンプト改良か、構文解説の範囲を狭める(例: 1題につき解説は重要3文のみ)
- 自分が3日続かなかった → 継続設計(ストリーク・レベル)を先に強化。題数を増やすのは後
